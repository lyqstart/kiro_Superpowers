# Parallel Agents Safety Policy

版本：v0.7.0

## 定位

本文件把 Superpowers 的 `dispatching-parallel-agents` 思想改造成 Kiro 的安全并行规则。目标是提速，但不能牺牲规格一致性、代码安全和可验证性。

## 默认策略

- 默认允许并行：context gathering、只读分析、spec review、code review、test review。
- 默认不并行：implementation、数据库迁移、API 契约修改、状态机修改、跨模块重构。
- 边界不清晰时，必须禁止并行。
- 并行不是目标，稳定交付才是目标。

## 允许并行的安全条件

只有同时满足以下条件，才允许并行执行：

1. 不改同一批文件。
2. 不改同一个接口契约。
3. 不改同一张数据库表。
4. 不共享迁移脚本。
5. 不共享状态机。
6. 每个并行任务都有独立验证命令。
7. 每个任务的输入、输出、完成定义清楚。
8. 每个任务可以独立失败，不影响其他任务回滚或判断。

任何一条不满足，结论必须是：禁止并行。

## Parallel Dispatch Plan

并行前必须输出计划，格式如下：

```text
Parallel Dispatch Plan

Task A:
- 目标：
- 涉及文件：
- 涉及接口：
- 涉及数据库表：
- 涉及状态机：
- 验证命令：
- 使用 agent：
- 风险：

Task B:
- 目标：
- 涉及文件：
- 涉及接口：
- 涉及数据库表：
- 涉及状态机：
- 验证命令：
- 使用 agent：
- 风险：

Conflict Check:
- 共享文件：无 / 有，列出
- 共享接口：无 / 有，列出
- 共享数据库表：无 / 有，列出
- 共享迁移脚本：无 / 有，列出
- 共享状态机：无 / 有，列出

Decision:
- 允许并行 / 禁止并行
- 理由：
```

不要省略 Conflict Check。没有 Conflict Check，就不能并行。

## 并行结束后的统一收口

并行任务结束后，Kiro main agent 必须汇总，而不是让各 subagent 各自宣布完成：

1. 汇总每个 agent 的结果。
2. 检查是否产生冲突。
3. 检查是否有共享文件被不同任务修改。
4. 统一进入 `sp-spec-reviewer`。
5. 统一进入 `sp-code-reviewer`。
6. 统一进入 `sp-test-verifier`。
7. 只有统一审查和验证通过，才能继续标记完成。

## 禁止行为

- 不允许为了提速而并行改同一个 API。
- 不允许并行改同一张数据库表或迁移脚本。
- 不允许并行实现边界不清的任务。
- 不允许 subagent 自己决定扩大并行范围。
- 不允许并行结束后跳过统一 review。

## 用户体验

用户不需要说“请并行”。当任务天然适合并行时，Kiro 可以提出并行计划；当风险存在时，Kiro 应该明确说“不并行更安全”。
