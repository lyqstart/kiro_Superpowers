# Systematic Debugging

## 调试四阶段

### 1. 根因调查

- 读完整错误信息。
- 稳定复现。
- 检查最近变更。
- 多组件系统要在边界加诊断信息。
- 沿数据流追踪坏值来源。

### 2. 模式分析

- 找同项目中能工作的类似代码。
- 对比差异。
- 理解依赖、配置、环境假设。

### 3. 假设验证

- 一次只提出一个假设。
- 用最小改动验证。
- 验证失败就回到调查，不叠加猜测。

### 4. 最小修复

- 写失败测试复现问题。
- 修根因，不修症状。
- 运行原 bug 验证和回归验证。

## 停止条件

如果连续 3 次修复失败，停止继续 patch，重新审查架构或需求。

## v1.3 深度调试增强

调试不是“先猜一个补丁”。当任务是 bugfix、失败测试、线上异常、异步不稳定、跨组件错误时，必须应用：

1. root-cause-tracing：从可观察错误追踪到根因，说明输入、数据变化、异常组件、触发条件和根因证据。
2. defense-in-depth：判断是否需要多层防护，覆盖输入、边界、外部依赖、异常状态和契约假设。
3. condition-based-waiting：涉及异步、队列、后台任务、文件生成、网络请求或测试等待时，禁止把固定 sleep 作为主要方案。
4. multi-component diagnostics：跨前端、后端、数据库、缓存、任务队列、外部 API 时，必须按组件边界诊断。
5. 3-fix architecture stop gate：同一问题连续三次修复失败时，停止继续打补丁，重新质疑需求、根因、架构假设和验证方式。
